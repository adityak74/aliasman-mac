<objective>
Implement the first complete user-facing alias workflow: explicit `add`, `update`, `delete`, and `list` commands through named CLI flags with validation, canonical metadata persistence, regenerated managed alias output, shell-specific reload hints, compact list output, and actionable errors.
</objective>

<requirements>
- **ALS-01:** User can add an alias with named flags.
- **ALS-02:** User can update an existing alias with named flags.
- **ALS-03:** User can delete an alias by name.
- **ALS-04:** User can list aliases in a readable CLI table.
- **ALS-05:** User can see useful errors for duplicate aliases, missing aliases, invalid flags, and write failures.
</requirements>

<threat_model>
| Threat | Severity | Mitigation |
|--------|----------|------------|
| CRUD command silently overwrites an existing alias | high | `add` must reject duplicates; `update` is the explicit replacement path. |
| Protected command alias is created accidentally | medium | Reuse `validate_alias_name_for_write` and require `--force`. |
| User sees success but current shell cannot use alias | medium | Every mutation prints a reload hint containing `source ~/.aliases`. |
</threat_model>

<truths>
- CRUD commands are explicit verbs with named flags: `add`, `update`, `delete`, and `list`.
- `add` never overwrites duplicates; duplicate errors point to `update`.
- Default `list` output is a compact table with `Name`, `Command`, and `Source`.
- CRUD errors include actionable next-step guidance.
- Every mutation prints shell-specific reload guidance and mentions opening a new terminal.
</truths>
